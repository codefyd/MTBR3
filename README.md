# MTBR3 — النسخة النظيفة المعتمدة

## إصدار SaaS 3.0

أضيفت ترقية متعددة الجمعيات تشمل عزل البيانات، المستخدمين وصلاحيات الصفحات، إدارة الاشتراكات، وبوابة MCP مستقلة لكل جمعية أو مستخدم.

للمشروع الحالي اقرأ `SAAS_SETUP.md` ثم شغّل `supabase/saas_upgrade.sql` بعد ملفات Campaign Intelligence حتى v2.4.

لا تشغّل `schema.sql` من جديد على قاعدة المشروع الحالية.

هذه نسخة تنظيف واعتماد للمشروع.

## محتويات النسخة

- صفحات HTML كما هي من النسخة المرفوعة.
- ملفات `assets/app.js` و`assets/config.js` و`assets/styles.css` كما هي من النسخة المرفوعة.
- ملف Supabase واحد معتمد: `supabase/schema.sql`.
- ملف إصلاح اختياري للمشاريع القائمة: `supabase/repair_existing_project.sql`.
- ملفات شرح مختصرة بدل ملفات التحديثات القديمة المتفرقة.

## التشغيل على مشروع جديد

1. أنشئ مشروع Supabase جديد.
2. افتح: Supabase Dashboard → SQL Editor → New Query.
3. انسخ كامل محتوى `supabase/schema.sql` وشغّله مرة واحدة.
4. عدّل `assets/config.js` إذا تغيّر رابط Supabase أو المفتاح العام.
5. ارفع ملفات الواجهة إلى GitHub Pages أو الاستضافة الثابتة.
6. أنشئ مستخدمًا من Supabase Authentication، ثم سجّل الدخول من `login.html`.

## التشغيل على مشروع قائم

إذا كانت القاعدة موجودة مسبقًا، لا تشغّل ملف `schema.sql` قبل أخذ نسخة احتياطية.
استخدم `supabase/repair_existing_project.sql` فقط عند الحاجة لإصلاح:

- تعارض دالة `donor_rebuild_chunk`.
- وجود قيد قديم على `operation_no` وحده.
- الحاجة لتثبيت المفتاح الصحيح للعمليات: `(line_no, operation_no)`.

## ملاحظات مهمة

- المفتاح الصحيح لمنع تكرار رفع نفس صف العملية هو: `(line_no, operation_no)`.
- لا تعتمد النسخة النظيفة على `operation_no` وحده؛ لأن العملية الواحدة قد تحتوي أكثر من صف.
- بعد رفع بيانات كثيرة، اترك خيار إعادة احتساب ملفات المتبرعين معطّلًا ثم شغّل إعادة الاحتساب يدويًا بعد انتهاء الرفع.
- للمشروع الجديد، غيّر `SUPABASE_URL` و`SUPABASE_KEY` في `assets/config.js` إذا كان مشروع Supabase مختلفًا.


## رفع العمليات — أسماء الأعمدة المقبولة

صفحة `operations.html` تقبل الآن أسماء بديلة لبعض الأعمدة عند رفع ملف CSV:

- المشروع: يقبل `المشروع` أو `البيان`.
- كود الإحالة: يقبل `كود الإحالة` أو `كودالإحالة` أو `الإحالة#` أو `#الإحالة` أو `الإحالة`.

## ذكاء الحملات التسويقية v2

أصبحت `campaign-analysis.html` منصة لإدارة الحملة وتحليلها، وليست مجرد جدول للأكواد. تدعم:

- حملات قصيرة أو مستمرة أو موسمية، مع قناة مستقلة مثل واتس اب أو Google أو منصة X.
- الربط بكود محدد، عدة أكواد، بادئة كود، مشروع أو عدة مشاريع.
- الجمع بين مجموعات الشروط بصيغة «و / أو»، مع استثناء أكواد ومشاريع.
- معاينة العوائد والعمليات والمتبرعين المطابقين قبل حفظ الحملة.
- تكاليف متعددة ومؤرخة للحملة بدل تكلفة واحدة ثابتة للكود.
- مؤشرات العائد، ROAS، نسبة التكلفة، صافي العائد، ومتوسط وأكبر تبرع.
- المتبرعون الجدد والسابقون والمتكررون والتبرعات اللاحقة للحملة.
- المستهدفون، الاستجابة خلال 24 ساعة و3 و7 أيام، ومتوسط زمن الاستجابة.
- الاحتفاظ بتقرير أكواد الإحالة السابق داخل تبويب مستقل.

لترقية المشروع القائم شغّل مرة واحدة في Supabase SQL Editor:

```text
supabase/campaign_analysis_v2.sql
```

ملف `campaign_analysis.sql` باقٍ للتحليل القديم فقط، ولا يلزم تشغيله إذا كان مثبتًا سابقًا.

### تحسين الأداء v2.1

بعد تثبيت الإصدار v2 شغّل مرة واحدة:

```text
supabase/campaign_analysis_v2_1_performance_fix.sql
```

يعيد الملف بناء دالتي القائمة والتقرير باستعلامات أخف، ويضيف الفهارس اللازمة دون حذف أو تعديل بيانات التبرعات.

### طبقة العمليات السريعة v2.2

للقواعد الكبيرة شغّل بعد ذلك:

```text
supabase/campaign_analysis_v2_2_cache_fix.sql
```

ينشئ ملخصًا واحدًا لكل رقم عملية ويُبقيه متزامنًا تلقائيًا مع عمليات الرفع والتعديل والحذف، وبذلك لا يعيد تحليل الحملات تجميع جدول العمليات كاملًا عند كل فتح.

### ذاكرة نتائج الحملات v2.3

إذا اكتمل v2.2 وتساوى عدد العمليات مع عدد الملخصات، لكن استمر خطأ `statement timeout` عند فتح قائمة الحملات، شغّل مرة واحدة:

```text
supabase/campaign_analysis_v2_3_result_cache.sql
```

قد يستغرق التشغيل الأول وقتًا لأنه يبني تقارير الحملات الحالية داخل SQL Editor. بعد ظهور `Success` تصبح القائمة والتقارير قراءات سريعة من ذاكرة نتائج، ويعيد زر «تحديث التحليل» بناء الحملات القديمة بالتتابع. لا يغيّر الملف بيانات العمليات أو المتبرعين أو المستهدفين.

### إصلاح الاستهداف والحذف والمهلة v2.4

بعد v2.3 شغّل مرة واحدة:

```text
supabase/campaign_analysis_v2_4_targets_delete_timeout_fix.sql
```

يمنع تعارض المفتاح الأجنبي عند حذف حملة، ويجعل تغييرات `campaign_targets` توسم نتائج الحملات للتحديث، ويرفع مهلة دالة تحديث الحملة الثقيلة وحدها إلى 55 ثانية دون تغيير المهلة العامة للمشروع.
