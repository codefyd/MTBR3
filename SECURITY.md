# دليل الحماية وإعداد المفاتيح (Security)

## فهم أساسي مهم
هذا مشروع **HTML ثابت يعمل في متصفح المستخدم** (GitHub Pages). أي مفتاح يصل للمتصفح يكون **مكشوفًا بطبيعته** — لا يمكن إخفاؤه. لذلك:

- **المفتاح العام (publishable / anon)** مصمّم ليكون علنيًا، ووجوده في الكود آمن.
- **حمايتك الحقيقية = تسجيل الدخول (Supabase Auth) + سياسات RLS** في قاعدة البيانات. هذا ما يمنع أي شخص من قراءة أو تعديل البيانات بدون حساب.
- **لا تضع أبدًا** مفتاح `service_role` السرّي في أي ملف HTML/JS — لأنه يتجاوز كل الحماية. هذا المشروع لا يحتاجه إطلاقًا.

---

## 1) إضافة المفاتيح في GitHub Secrets
**Settings → Secrets and variables → Actions → New repository secret**، أضف:

| الاسم | القيمة |
|------|--------|
| `SUPABASE_URL` | `https://gaermmsfbtfcghegdkmq.supabase.co` |
| `SUPABASE_KEY` | المفتاح العام (publishable) |

ملف `.github/workflows/deploy.yml` يحقنها تلقائيًا في `assets/config.js` وقت النشر، فلا تبقى مكتوبة في المستودع. (تبقى ظاهرة في المتصفح بعد النشر — وهذا طبيعي وآمن للمفتاح العام.)

### تفعيل النشر
**Settings → Pages → Build and deployment → Source: GitHub Actions**. بعدها كل دفع لفرع `main` ينشر الموقع تلقائيًا.

---

## 2) تشغيل Code scanning (Security and quality)
**Settings → Code security → Code scanning → Set up → Default** (أو CodeQL مع إعداد "Security and quality"). هذا يفحص الكود تلقائيًا. ملاحظة: قد تظهر تنبيهات بأن المفتاح العام مكشوف — وهي تنبيهات يمكن تجاهلها بأمان للمفتاح *العام* فقط (وثّق ذلك في المستودع).

كذلك فعّل: **Secret scanning** و **Push protection** و **Dependabot alerts** من نفس الصفحة — تمنع رفع مفاتيح سرّية بالخطأ.

---

## 3) تعزيز الحماية في Supabase (مهم)
في **Authentication → Providers / Sign In** :
- **عطّل التسجيل الذاتي** (Disable sign-ups) ليبقى النظام للإدارة فقط؛ أنشئ الحسابات يدويًا من **Authentication → Users**.
- فعّل **كلمة مرور قوية** و(إن أمكن) المصادقة الثنائية للحسابات.

في **Authentication → URL Configuration** :
- ضع **Site URL** = رابط موقعك على Pages، وأضف نفس الرابط في **Redirect URLs**.

تأكّد أن **RLS مفعّل** على كل الجداول (ملف `schema.sql` يفعّله) — هذا هو خط الدفاع الأساسي.

---

## 4) تطبيع أرقام الجوال (تنظيف تلقائي عند الرفع)
النظام الآن يصحّح الأرقام تلقائيًا داخل قاعدة البيانات وفي المتصفح بنفس المنطق:

| الحالة | المُدخل | الناتج |
|-------|--------|--------|
| سعودي 9 خانات | `599992509` | `966599992509` |
| سعودي بصفر | `0551234567` | `966551234567` |
| سعودي كامل | `966532176617` | كما هو |
| أجنبي | `201210673447` / `447831275999` | كما هو |
| مبتور/خاطئ | `5530538` (7 خانات) | **يُرفض** |
| مشغّل غير صحيح | `5XXXXXXXX` بمشغّل خارج (50,53–59) | **يُرفض** |

- **المشغّلات السعودية المقبولة:** 50, 53, 54, 55, 56, 57, 58, 59.
- بعد كل رفع، إن وُجدت أرقام مرفوضة تظهر **نافذة تقرير** فيها (الرقم كما ورد + سبب الرفض) مع زر **تصدير CSV** لمراجعتها وتصحيحها يدويًا ثم إعادة رفعها.

---

## 5) CSV فقط
كلاهما مدعوم. **CSV أسرع وأخف** للملفات الكبيرة (رفعًا وتصديرًا) ويُنصح به. النظام يقرأ عمود الجوال **كنص دائمًا**، فلا تتأثر الأرقام الطويلة (مثل `12109353928`) بمشكلة الصيغة العلمية التي تسببها برامج الجداول.
