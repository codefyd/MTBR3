# تحويل ولاء إلى SaaS متعدد الجمعيات

## النتيجة

- عزل كامل لبيانات كل جمعية بواسطة `organization_id` وسياسات RLS.
- مستخدمون بأدوار وصلاحيات ظهور صفحات.
- إدارة الجمعيات والبريد وتاريخ نهاية الاشتراك من `admin.html`.
- بوابة MCP للقراءة والتحليل بمفتاح مستقل لكل جمعية أو مستخدم.
- إلغاء مفتاح MCP فوراً، وتاريخ انتهاء، وسجل تدقيق لكل استدعاء.

## ترتيب التنفيذ على المشروع الحالي

1. خذ نسخة احتياطية من قاعدة البيانات قبل الترقية.
2. ارفع ملفات الواجهة الجديدة إلى نفس الاستضافة.
3. افتح Supabase → SQL Editor وشغّل كامل الملف:

   `supabase/saas_upgrade.sql`

4. انتظر ظهور `Success`. لا تغلق الصفحة أثناء التنفيذ، لأن الملف يعيد بناء Cache العمليات.
5. من Supabase → Edge Functions أنشئ وظيفتين:

   - `saas-admin` من `supabase/functions/saas-admin/index.ts` مع تفعيل Verify JWT.
   - `mcp` من `supabase/functions/mcp/index.ts` مع تعطيل Verify JWT؛ الوظيفة تتحقق بنفسها من مفتاح MCP المشفّر.

6. تأكد أن `assets/config.js` يحتوي رابط المشروع ومفتاحه العام فقط.
7. سجّل الدخول بأقدم حساب موجود في Supabase Auth؛ ملف الترقية يجعله مدير المنصة تلقائياً.
8. افتح «إدارة المنصة»، وعدّل «الجهة الحالية» إلى اسم الجمعية الحالية وحدد تاريخ اشتراكها.
9. أضف الجمعيات الأخرى ومستخدميها وصلاحيات صفحاتهم من الصفحة نفسها.

## ربط MCP

من «إدارة المنصة» اختر الجمعية ثم «إنشاء مفتاح MCP». سيظهر:

- رابط واحد من نوع Streamable HTTP.
- مفتاح `Authorization: Bearer` يظهر مرة واحدة فقط.
- زر ينسخ إعداد JSON جاهزاً للعملاء الذين يدعمون رؤوس HTTP المخصصة.

لا تضع المفتاح في عنوان URL أو ملف واجهة عام. إذا انكشف، ألغِه فوراً وأنشئ مفتاحاً جديداً.

بوابة MCP في هذا الإصدار للقراءة والتحليل فقط. الأدوات المتاحة:

- ملخص لوحة التحكم.
- البحث في المتبرعين وجلب ملف متبرع.
- البحث في العمليات.
- قائمة الحملات وتحليل حملة.
- قائمة المشاريع.

إذا كان المفتاح مربوطاً بمستخدم، فلن تظهر له أداة مرتبطة بصفحة محجوبة عنه.

## ملاحظات الاشتراك

- إيقاف الجمعية أو انتهاء تاريخها يمنع الوصول إلى بياناتها من الواجهة ومن MCP.
- حساب مدير المنصة يبقى قادراً على فتح `admin.html` لتجديد الاشتراك.
- الجمعية القديمة تبدأ دون تاريخ انتهاء حتى تعدله يدوياً من لوحة الإدارة.

## اختبار العزل بعد التنفيذ

1. أنشئ جمعيتين تجريبيتين ومستخدماً لكل جمعية.
2. ارفع عملية تجريبية بالرقم نفسه في الجمعيتين؛ يجب أن تُحفظ العمليتان دون تعارض.
3. سجّل الدخول بكل مستخدم وتأكد أنه لا يرى إلا بيانات جمعيته.
4. أخفِ صفحة «العمليات» عن مستخدم وتأكد من اختفائها ومنع فتح رابطها مباشرة.
5. أنشئ مفتاح MCP لكل جمعية ونفّذ `get_dashboard_summary`؛ يجب أن تختلف النتائج بحسب الجمعية.
6. ألغِ أحد المفاتيح وتأكد أن الطلب التالي يرجع `401`.
